# Deduplication

去重逻辑、ID生成和冲突解决策略。

## 核心原则

- 同一条日记不会重复进入 Raw Record
- 同一个想法不会重复进入 Pool
- 状态文件是唯一的真相来源

## Raw Record 去重

### 写入前检查

1. 读取目标 Raw 文件（如 `Raw2026-02.md`）
2. 搜索 `## 2026-MM-DD` 标题
3. 如果日期已存在 → 跳过该日期
4. 如果不存在 → 在文件末尾追加

### 示例

```
Raw2026-02.md 已有内容：
  ## 2026-02-25
  ## 2026-02-26
  ## 2026-02-27

新请求处理 2026-02-25 to 2026-02-28：
  - 跳过 02-25 至 02-27（已存在）
  - 追加 02-28（新内容）
```

## Pool 去重（ID机制）

### ID格式

| 类型 | ID格式 | 示例 |
|------|--------|------|
| Ideas | `idea-{日期}-{哈希}` | `idea-20260226-a3f7b2` |
| Seeds | `seed-{日期}-{哈希}` | `seed-20260207-b8c2d1` |
| Insights | `insight-{日期}-{哈希}` | `insight-20260226-f2a9e3` |

### 哈希生成

```python
# 伪代码
content_hash = hash(content)[:6]
item_id = f"{type}-{date}-{content_hash}"
```

### 写入前检查

1. 计算提取项的 ID
2. 检查 `.jr-state.json` 的 `extractedItems`
3. 如果 ID 已存在 → 跳过
4. 如果不存在 → 追加到 Pool 文件，并记录 ID

### Pool文件格式

**Ideas Backlog**:
```markdown
- [ ] #idea/content #工具 id:idea-20260226-a3f7b2
  - **想法：** ...
  - **来源：** [[2026-02-26]]
  - **添加日期：** 2026-02-28
```

**Content Seeds**:
```markdown
### Seed #202602260001 id:seed-20260226-b8c2d1
- **素材：** ...
- **来源：** [[2026-02-26]]
- **添加日期：** 2026-02-28
```

**Insights Pool**:
```markdown
### 2026-02-28 id:insight-20260226-f2a9e3
**Insight:** ...
- **来源：** [[2026-02-26]]
- **添加日期：** 2026-02-28
```

## 冲突解决

### 场景 1: 手动整月后，自动周触发

**问题**: 2月28日手动处理 2月1-28日，3月2日自动处理"上周"（2月24日-3月2日）时，2月24-28日会重复。

**解决**:
```
1. 自动模式检查状态文件
2. 发现 2026-02-24 至 2026-02-28 已标记为 processed
3. 跳过这些日期
4. 只处理 2026-03-01 和 2026-03-02
```

### 场景 2: 重复手动触发同一日期

**问题**: 用户多次手动处理同一日期范围。

**解决**:
```
1. 提示"检测到已处理日期，是否跳过？"
2. Raw Record: 搜索标题，已存在则跳过
3. Pool: 检查 ID，已存在则跳过
```

### 场景 3: Pool 文件手动编辑后 ID 丢失

**问题**: 用户在 Pool 文件中手动添加内容，没有 ID。

**解决**:
```
1. 运行时重新扫描 Pool 文件
2. 为没有 ID 的项生成新 ID
3. 更新状态文件
4. 未来以 ID 为准
```

### 场景 4: 跨月日期范围

**问题**: 处理 2026-02-25 到 2026-03-02（跨月）。

**解决**:
```
1. 处理 02-25 至 02-28 → Raw2026-02.md
2. 处理 03-01 至 03-02 → Raw2026-03.md
3. 状态文件中分别记录
```

## 重新处理（强制模式）

用户选择"重新处理全部"时：

1. **Raw Record**: 仍然检查标题，避免物理重复追加
2. **Pool**: 使用 ID 去重，不生成重复项
3. **Weekly Review**: 重新生成（覆盖或新建）
4. **状态文件**: 更新为新的 review 标记

## 临时输出模式

用户可选择"输出临时分析（不保存）"：

1. 生成分析内容
2. 显示给用户
3. 不写入任何文件
4. 不更新状态文件
5. 用户可选择"追加到 vault" → 进入正常流程
